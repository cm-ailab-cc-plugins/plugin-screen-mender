# Stage 5 — MR（發 MR / dry-run 產物）

- 對應 SKILL Phase 4 開 MR + [`orchestration`](../../skills/screen-mender/references/orchestration.md) §5。
- 一畫面一個 MR。

## Inputs
- `fix_record` / `verify_verdict` / `residual_visible` / wont-fix / `deferred`（前幾個 stage 的產物）。
- before·after 截圖、`worktree` / `branch` / `base_branch` / `mr_tool` / `unified_id` / `dry_run` / `run_dir`。

## dry_run=true（不開 MR）
- 不 rebase、不 push、不開 MR。
- 從 worktree 取產物落 `<run_dir>/<unified_id>/`：
  - `change.patch` = `git -C <worktree> format-patch <base_branch>..HEAD --stdout`。
  - `before.png` / `after.png`。
  - `proposed-mr.md` = 下面 description schema，但截圖引本地相對路徑（不 `POST /uploads`），轉 ready 改標 `would-be-ready`。
- 結束。

## dry_run=false
1. **冪等**：先 live 查 `<mr_tool> mr list`，該 branch（`<feature_branch_prefix><unified_id>`）已有 open MR → 跳過不重開。
2. **rebase + push**：rebase 到 `base_branch`（已 push 用 `--force-with-lease`），開一個 MR（一畫面一個）。
3. **description**（schema 見 [`issue-schemas`](../../skills/screen-mender/references/issue-schemas.md) §4）：
   - 標題固定模板 `自動跑版修復[（部分）]：<unified_id> - <原因摘要>`（有任一殘留可見 → **必用「（部分）」** variant）。
   - 畫面狀態：`fully-fixed` / `partially-fixed (n fixed, m deferred-visible)` / `clean`——由「所有 kept+deferred 是否都解決」計，非「我選修的那幾條過 verify 沒」。
   - 修了哪些缺陷（file:line + 修法；退讓解註記 `legibility-degraded`）。
   - ⚠️ 殘留可見缺陷（`deferred:needs-design`／`deferred:deferred-by-run-config`，after 圖仍可見）列最顯眼處。
   - 考慮過但不修（wont-fix reason）。
   - 內嵌 before/after：`POST /projects/:id/uploads`（multipart，`glab api -F` 不支援，用 `curl -F file=@<png>` + token）取 `/uploads/...` markdown 嵌入。
4. **轉 ready**：`verify_verdict` PASS（審查＋驗證一格已涵蓋 scope/redesign + AC + 視覺等價 + 無 regression）→ `<mr_tool> mr update <id> --ready`。

## Output
- `mr_url`（或 dry-run 的 `proposed-mr.md` 路徑）。
- 畫面狀態（`fully-fixed` / `partially-fixed (n,m)` / `clean`）。
- 誠實鐵則：after 圖只要還看得到缺陷，畫面就**不是 fully-fixed**，不論歸因。
  - 降 partially-fixed ＋ 殘留可見。
