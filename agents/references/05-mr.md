# Stage 5 — finalize（定稿 + 交出 section，**不開 MR**）

- 本階段只負責**把本畫面定稿**並交出彙整素材；MR 由 run 尾的 `screen-mender-integrator` 一次彙整全部畫面後開**單一** MR（見 [`06-integrate`](06-integrate.md)、orchestration §5）。
- 一個 run 一個 MR；本畫面**不 push、不 rebase、不開 MR**。
- per-screen branch 留在本地共用 .git（lane worktree 是 canonical repo 的 `git worktree`，refs 共用），整 run 末由 integrator `cherry-pick` 取用——故無需 push。

## Inputs
- `fix_record` / `verify_verdict` / `residual_visible` / wont-fix / `deferred`（前幾個 stage 的產物）。
- before·after 截圖、`worktree` / `branch` / `base_branch` / `unified_id` / `dry_run` / `run_dir`。

## Procedure（dry_run 與否共通）

1. **確認已 commit**：本畫面所有修復（含 snapshot test 檔）都已 commit 在 per-screen branch `<branch>` 上（fix 階段已逐步 commit；此處只確認 `git -C <worktree> status` 乾淨、無漏 add 的 test/資源檔）。
   - **不 push、不 rebase**（rebase/合併歸 integrator，避免多階段互踩）。
   - 一畫面一 commit（建議）：若 fix 階段產生多個 commit，於此 `git -C <worktree> reset --soft <base_branch>` 後重做**一個** commit，commit message 用固定模板（見下），讓 integrator cherry-pick 後 MR 內**一畫面一 commit**、reviewer 可逐 commit 審。
2. **交出彙整素材**到 `<run_dir>/<unified_id>/`：
   - `mr-section.md` = 本畫面的 MR 段落（schema 見 [`issue-schemas`](issue-schemas.md) §4〈per-screen section schema〉）。截圖一律引 **run_dir 內相對檔名**（`before__<state>__<locale>.png` / `after__...`）——integrator 開 MR 時才 `POST /uploads` 換成 `/uploads/...`；dry_run 則保留相對路徑。
   - `before__<state>__<locale>.png` / `after__...`（capture/verify 階段已落此，確認齊備）。
   - `meta.json`：`{ unified_id, branch, status, base_branch, build_cmd, snapshot_test_cmd }`，供 integrator 知道該 cherry-pick 哪條 branch、標哪個狀態、衝突時用哪些指令 build/重跑 test。

## commit message 固定模板（一畫面一 commit）

```
自動跑版修復[（部分）]：<unified_id> - <原因摘要>
```

- 與 MR 標題模板同字彙（[`issue-schemas`](issue-schemas.md) §4）；有任一殘留可見 → 用「（部分）」variant。
- body 簡列修了哪幾條（一行一條），詳細描述留 `mr-section.md`。

## Output（runner return 用，精簡）
- `section_path` = `<run_dir>/<unified_id>/mr-section.md`（**取代舊的 `mr_url`**；MR 由 integrator 開）。
- 畫面狀態（`fully-fixed` / `partially-fixed (n,m)` / `clean`）。
- 誠實鐵則：after 圖只要還看得到缺陷，畫面就**不是 fully-fixed**，不論歸因。
  - 降 partially-fixed ＋ 殘留可見。

> 冪等不在本階段做（不再 per-branch 查 open MR）：已修畫面在 audit 階段自然回 `clean`（無新缺陷）→ 不產 section、不被 cherry-pick；run 級冪等靠 integration branch 名含 `run_id`（見 [`06-integrate`](06-integrate.md)）。
